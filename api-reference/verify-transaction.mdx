---
title: 'Verify Transaction'
api: 'GET https://api.rolla.io/api/v1/transaction/verify/{reference}'
description: 'Verify the status of a transaction using its reference'
---

## Authentication

All requests must include your secret API key in the Authorization header:

```
Authorization: Bearer sk_live_your_secret_key
```

<Note>
  The organization is automatically determined from your API key. You don't need to provide an organization ID.
</Note>

## Path Parameters

<ParamField path="reference" type="string" required>
  The unique transaction reference returned when the transaction was initiated (e.g., `TXN_gGbEEvK25eVgmgrf`)
</ParamField>

## Transaction Statuses

| Status | Description |
|--------|-------------|
| `pending` | Transaction has been created but not yet processed |
| `requires_action` | Additional action required (3DS, PIN, OTP) - customer has not completed authentication |
| `processing` | Transaction is being processed by the payment provider |
| `successful` | Transaction completed successfully - you can fulfill the order |
| `failed` | Transaction failed - do not fulfill the order |

## Best Practices

<Warning>
  Always verify the transaction status on your server before fulfilling orders. Never rely solely on client-side callbacks.
</Warning>

1. **Verify after callback**: When your customer is redirected back to your `callbackUrl`, always call this endpoint to verify the transaction status.

2. **Verify before fulfillment**: Before delivering goods or services, confirm the transaction status is `successful`.

3. **Handle pending states**: If status is `requires_action` or `processing`, wait and poll again after a few seconds.

4. **Use webhooks**: For real-time updates, configure [webhooks](/api-reference/webhooks) to receive transaction status notifications.

## Alternative: POST Method

This endpoint also supports POST requests with the same path:

```bash
POST /api/v1/transaction/verify/{reference}
```

Both methods return the same response.
